1. Introduction

Communication started since the beginning of human creation as a way to exchange messages. Distant communication began as early as the 18th century with the introduction of television, telegraph, and then telephony. The emergence of computer network and telecommunication technologies bear the same objective, that is to allow people to communicate. Teleconferencing or Chatting is a method of using technology to bring people and ideas together despite of the geographical barriers. This project is to create an easy-to-use web application that allow an instant messaging solution to enable users to seamlessly communicate with each other. This section goes into further detail on the product scope, target audiences, overview, acronyms used, and references to standards and best practices.

1.1 Document Purpose

Purpose of this software requirement specification document is to provide a complete description of the features which will be implemented in this social platform project. Moreover, this document includes user scenarios, working principles of the system, and internal and external interfaces. Intended audiences are users and the development team. Use case diagrams will provide visual description for users and use case narrative will provide technical information for developers. Also, tests can be defined according to use case descriptions in this document.

1.2 Product Scope

Our application will be on mobile devices, to allow accessibility to a larger population since cellphones are used by a large and diverse audience, from youngsters to elders. The software will be easy to use and synchronized with the user’s contacts. This will be an automatic process when the user downloads the software. The user will only be able to send messages to people in his contact list that also have downloaded the software and have registered themselves. Some basics feature the software will incorporate are: creating groups, sending text messages, images, videos, audios and other types of files.

2. Overall Description

2.1 Product Perspective

TheConnector is a Firebase web-chat application which features functionalities similar to most communication applications; this includes managing chats and creating a network of friends. TheConnector will store user interactions and users’ information into a relational MySQL database to be constantly updated and fetched by the user. Furthermore, TheConnector will be built upon an open source web-chat SDK that will require TheConnector to be a Google Firebase application. Lastly (and optional), TheConnector will be upholding an E2EE standard that will encourage the application to interact with the Virgil SDK, Virgil Crypto Library, Virgil e3kit SDK, and Virgil Cloud.

2.2 Product Functionality

TheConnector will have the following core functionalities:

  • Authentication Module:
    • Allow users to register
    • Allow users to reset password
    • Allow users to login
    • Allow users to logout
  • Chat Module:
    • Users can create new chat
    • Users can delete chat
    • Users can send messages
  • Friend list Module:
    • Allow users to send friend request
    • Allow users to accept/decline friend request
    • Allow users to unfriend another user
  • Account Module:
    • Allow users to update account information

2.4 Design and Implementation Constraints

Web-Chat SDK:

  • SDK requires TheConnector web-chat application to be a Google Firebase application.
  • SDK is dependent on Firebase CLI, thus additional configurations and prerequisites are required.
  • SDK is open source and managed by third party; bugs and issues are resolved through Chat21.

MySQL:

  • MySQL database cannot store images, videos, or files; TheConnector will be restricted to sending text messages as a result.
  • Saved text messages have a 4,294,967,295-character limit.
  • Database must be reliable, dependable, and efficient.

2.6 Assumptions and Dependencies

  • TheConnector app relies on Chat21 open source web-chat SDK, which requires TheConnector to be a Firebase application. If Google Firebase experiences a massive update but, the Chat21 development team neglects to update their SDK, then TheConnector will encounter bugs. Depending on the severity of these bugs and the responsiveness of Chat21 to resolve these issues, TheConnector development team will have to potentially switch to a new web-chat SDK with different configurations and prerequisites.
  • TheConnector is a Firebase web application: users must be connected to the internet and using a web browser that is support by Google Firebase applications (Google Chrome, Firefox, Safari and Edge).

3. Specific Requirements

3.1 External Interface Requirements

3.1.1 User Interfaces

Main Launch Screen:

Upon launching TheConnector web-app, a login screen will appear, prompting the user to input their username and password into the text fields provided. Beneath the text fields are two buttons: “login” and “register”. Next to the buttons is a link for users who forgot their password.

Registration Screen:

Upon clicking “register”, the user is taken to a registration page where the following information is asked and collected: first name, last name, username, email, confirmed email, password, confirm password. A “submit” button is located at the bottom of the screen for users to submit their online registration form once they have completed it.

Reset Password Screen:

Upon clicking the link “Forgot Password?”, the user is taken to a page where they can reset their password. The following data is collected before the user can hit a button to send an authentication code to their email: username and email. Once the user receives the authentication code, they may input it into a textbox, click submit, and proceed to the next step to resetting their password.

Finish Resetting Password Screen:

Upon submitting an authentication code, the user is brought to a new page where they may continue resetting their password. The user inputs a new password, confirms the new password and clicks “reset password” button to change their password.

Login Screen:

Upon clicking the login button the user is brought to their profile page.

Logout Screen:

Upon clicking the logout button the user is brought back to the login page.

General Frame of TheConnector App:

The general frame of TheConnector will remain visible and active while the user is logged in. The general frame provides the user with a navigation tool that allows them to visit their chats, friend list, notifications, and profile page.

Profile Screen:

The user’s profile page will display their profile picture and username. Below the profile username will be an option to logout of the application. Besides the profile picture is an option to update account information.

Friend List Screen:

The friend list tab allows the user the ability to create new chats with those who are in their friend list by clicking on the chat bubble icon next to their friends’ names. Furthermore, the friend list tab provides a shortcut to viewing friends profiles. Lastly, users can find new/more friends to invite through typing their usernames into the search bar.

Other User Profile Screen:

Other account profile screens will look the same as user’s profile screen, but with an “add friend” icon below the username.

Friend Profile Screen:

Users have the option to unfriend someone by navigating to their friend’s profile page and clicking the “unfriend” button.

Chat Screen:

The chat screen displays the active chats in a panel to the left, while displaying the messages of the chat in a panel to the right. There is also a search bar to allow users to search up an active chat given the name of a friend. Upon selecting a chat, the user has the option to delete the chat by clicking on the “trash can” icon.

Notification Screen:

The notification screen displays the following: user friend requests, friend invites, and those who accepted the user’s friend request. The user has the option to accept or decline a friend invite.

3.1.3 Software Interfaces

Chat21 Open Source Web-Chat SDK

TheConnector will be using an open source web-chat SDK developed by Chat21 to integrate instant messaging components. Due to Chat21’s SDK implementation, TheConnector will need to be a Google Firebase application. The setup requirements needed to integrate Chat21 web-chat SDK are as follows:

Pre-requisites:

  • NPM installation: Used to manage dependencies and connect with Firebase CLI.
  • NodeJS installation: Used to connect with Firebase CLI.
  • Create a Firebase Project.
  • Install Firebase CLI.

Project Setup:

  • Clone or download Chat21 repository from GitHub.
  • Follow Chat21 chat SDK installation guide.
  • Login into Firebase CLI with Firebase login
  • Set up your Firebase project as instructed by Chat21, select Project ID and follow the instructions.

Project Deployment

  • http://www.chat21.org/docs/platform/
  • https://github.com/chat21/chat21-cloud-functions#pre-requisites

Friend List and Relational Model

TheConnector offers users a friend network building system. Friend invites will be programmed using JAVA and friendship relationships will be stored in MySQL relational database through SQL queries.

3.1.4 Communications Interfaces

Encryption Standards (Optional)

TheConnector Firebase web-chat application will be using End-to-End Encryption standard to ensure users’ messages are encrypted at the source and decrypted upon message receival. E2EE will be integrated into TheConnector app through an open source third party, Virgil Security.

The following components are used to integrate Virgil Security’s E2EE method:

  • TheConnector needs to provide users with a JWT that contains the user's identity, which is a string that uniquely identifies each user in the application.
  • Virgil Security’s VirgilSDK and Virgil-Crypto libraries will be used to generate and sign the JWTs on the server end.
  • On the client end, the Virgil Security’s e3kit SDK will be used to create and store the user's private key on their device and publish the user's corresponding public key in the Virgil Cloud, as well as providing digital signatures to verify data integrity.

4. Other Non-Functional Requirements

4.1 Performance Requirements

  • The application will be able to support at least 10,000 simultaneous users. The capacity can be extended in the future if needed.
  • The application will be dealing with data obtained from account updates, instant messaging, and friend network building, which will require a relational database.
  • Deleting a chat, friending/unfriending a user, and account updates should be displayed within the next refresh of the page.
  • All the functionality that the application will perform should be executing at a speed of no more than 5 seconds.

4.2 Safety and Security Requirements

  • Authentication cannot be hacked.
  • Data must be secured.
  • Message encryption and encrypted connections must be up to industry standards: VPN (virtual private network), SSL(secure socket layer), TLS(transport layer security), etc.
  • Application is resistant to malware attacks.

4.3 Software Quality Attributes

  • 4.3.1 Reliability
    • Software should keep the database information consistent.
    • TheConnector graphical user interface should not crash.
    • Database failures should be minimal and there should be crash recovery system in order to not lose information in a potential database failure.
  • 4.3.2 Availability
    • System should be available 7 days a week and 24 hours a day.
    • On the application side, system should be tested against probable failures before publishing the final version or updating a newer version of the application.
    • Published version should be error free.
  • 4.3.3 Maintainability
    • Upgraded versions of the application should be easily maintained for development; it should be easy to add code to the existing system, upgrade new features, and integrate other software components.
    • Maintenance should be cost effective and easy.
  • 4.3.4 Portability
    • TheConnector is a Firebase web chat application and will be accessible through different browsers such as Google Chrome, Firefox, Safari and Edge.
  • 4.3.5 Usability
    • The app password must contain at least 8 characters that includes at least one lower-case letter and one upper-case letter. The password must also contain a special character.
    • The user name cannot be empty and should be unique
    • The application should be simple to navigate and easy to learn.
    • Easy to use for input preparation, operation, and interpretation of output.
    • Provide consistent user interface standards or conventions with our other frequently used systems.
    • Easy for new or infrequent users to learn to use the system.
